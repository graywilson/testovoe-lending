# Вадим Искаков — лендинг-портфолио

Небольшой, но «полноценный» лендинг-презентация разработчика: современный
адаптивный фронтенд, рабочая форма обратной связи с отправкой писем и тремя
AI-функциями. Сделан как демонстрация **полного цикла**:
`frontend → API → обработка ошибок → результат`.

> 🔗 **Демо:** https://vadim.moimoment.ru/
> 💻 **Репозиторий:** https://github.com/graywilson/testovoe-lending

---

## TL;DR — запуск одной командой

На сервере с Docker (например, Ubuntu 24.04) достаточно перенести папку проекта
и выполнить:

```bash
docker compose up -d --build
```

Сайт поднимется на **порту 41321** → `http://<ip-сервера>:41321`.
Фронтенд и API работают из одного контейнера на одном порту.

> Реальные секреты лежат в файле `.env` (он в `.gitignore`, но переносится вместе
> с папкой). Если разворачиваете из git-клона — скопируйте `.env.example` в `.env`
> и заполните значения.

## Стек

**Frontend**
- TypeScript (строгий режим)
- Vite (сборка и dev-сервер)
- SCSS (модульная архитектура `@use`/`@forward`, без фреймворков)
- Без UI-библиотек — вёрстка и интерактив руками: адаптив, доступность,
  `IntersectionObserver`, работа с `fetch` и SSE-стримингом

**Backend**
- Node.js 20 + Express + TypeScript
- `zod` — валидация входных данных
- `nodemailer` — отправка писем (SMTP)
- `express-rate-limit` — защита публичных эндпоинтов
- Нативный `fetch` для запросов к AI (без лишних SDK)

**AI**
- Groq API (OpenAI-совместимый), модель `openai/gpt-oss-120b`

**Инфраструктура**
- Docker (multi-stage build) + docker-compose, один контейнер, порт 41321

├── server/                     # Бэкенд (Express + TS)
│   ├── src/
│   │   ├── index.ts            # Сервер: middleware, роуты, статика, ошибки
│   │   ├── config.ts           # Чтение .env, значения по умолчанию
│   │   ├── routes/             # contact.ts, ai.ts
│   │   ├── services/           # groq.ts, mailer.ts, emailTemplates.ts
│   │   ├── data/profile.ts     # Контекст о Вадиме для AI-ассистента

├── docker-compose.yml          # Запуск одной командой (порт 41321)

Принцип: **данные отделены от разметки** (секции «Стек» и «Проекты» рендерятся
из `content.ts`), **валидация и AI вынесены в сервисы**, **секреты — только на сервере**.

## Как реализована форма

Поля: **имя, телефон, email, комментарий**. Полный путь обработки:

1. **Клиент** (`modules/form.ts`)
   - Валидация на лету (blur + ввод), подсветка невалидных полей, понятные тексты ошибок.
   - Состояния кнопки: `loading` (спиннер) → `success` / `error` (баннер под формой).
   - Все ошибки сети/сервера обрабатываются и показываются пользователю.

2. **Запрос** → `POST /api/contact` (относительный путь; в dev проксируется Vite).

3. **Сервер** (`routes/contact.ts`)
   - Повторная валидация через `zod` (сервер — источник правды). На ошибки — `400`
     с картой ошибок по полям, которые фронт раскладывает обратно к инпутам.
   - **AI-анализ заявки** (необязательный бонус) — короткое саммари, категория,
     приоритет и подсказка для ответа. Если AI недоступен — тихо пропускается.
   - **Отправка двух писем** через `nodemailer`:
     - владельцу сайта (`OWNER_EMAIL`) — уведомление + AI-анализ, `Reply-To`
       проставлен на email заявителя;
     - пользователю — копия-подтверждение «заявка принята» с его данными и контактами.
   - Письмо владельцу критично (его сбой = ошибка запроса), копия отправляется
     «best-effort».

4. **Ответ** → фронт показывает success/error.

Письма свёрстаны на таблицах с инлайновыми стилями (`services/emailTemplates.ts`) —
тёмный градиентный хедер в стилистике сайта + светлая читаемая карточка, чтобы
корректно отображаться в Gmail/Outlook/Mail.ru.

## AI-функции (и как они устроены)

Ключ Groq используется **только на сервере** — браузер ходит к нашему API, ключ
в бандл не попадает.

1. **«Волшебная палочка» у поля комментария** — `POST /api/ai/improve`
   Переписывает черновик пользователя в аккуратное деловое письмо, сохраняя смысл.
   Есть откат к исходному тексту, состояние загрузки и обработка ошибок.

2. **AI-анализ входящей заявки** — внутри `POST /api/contact`
   Перед отправкой письма владельцу модель формирует саммари, категорию,
   приоритет и подсказку «с чего начать ответ». Результат — отдельный блок в письме.
   Это пример полезного AI в рабочем процессе (триаж входящих обращений).

3. **Плавающий AI-ассистент** — `POST /api/ai/chat` (**стриминг через SSE**)
   Отвечает на вопросы рекрутёра о стеке, опыте и проектах. Контекст о Вадиме
   задаётся system-промптом на сервере (`data/profile.ts`). Ответ печатается
   токен за токеном; есть индикатор «печатает…», подсказки-вопросы, отмена запроса
   при закрытии окна и обработка ошибок.

## Запуск

### Вариант 1 — Docker (рекомендуется)

Требуется Docker с плагином Compose.

```bash
# из папки проекта
docker compose up -d --build      # собрать и поднять в фоне
docker compose logs -f            # смотреть логи
docker compose down               # остановить
```

→ `http://localhost:41321` (или IP сервера).

### Вариант 2 — локальная разработка (без Docker)

Нужен Node.js 20+. Запускает бэкенд (41321) и фронтенд (Vite, 5173) параллельно;
Vite проксирует `/api` на бэкенд.

```bash
npm run setup     # установить зависимости в корне, server/ и client/
npm run dev       # бэкенд + фронтенд одновременно
```

→ фронтенд на `http://localhost:5173`, API на `http://localhost:41321`.

### Вариант 3 — продакшн-сборка локально

```bash
npm run build     # собрать фронт и бэк
npm start         # сервер отдаёт статику + API на 41321
```

## Переменные окружения (`.env`)

| Переменная | Назначение | По умолчанию |
|---|---|---|
| `PORT` | Порт приложения | `41321` |
| `SMTP_HOST` / `SMTP_PORT` / `SMTP_SECURE` | Параметры SMTP | `smtp.beget.com` / `465` / `true` |
| `SMTP_USER` / `SMTP_PASS` | Учётка SMTP | — |
| `MAIL_FROM_NAME` / `MAIL_FROM_ADDRESS` | Поле «От кого» | Вадим Искаков / `info@mailmoment.ru` |
| `OWNER_EMAIL` | Куда падает уведомление о заявке | `graywilsonwise@gmail.com` |
| `GROQ_API_KEY` | Ключ Groq (включает AI-функции) | — |
| `GROQ_MODEL` | Модель | `openai/gpt-oss-120b` |

> Если `GROQ_API_KEY` не задан — сайт и форма работают, AI-функции аккуратно
> отключаются (письмо уходит без AI-анализа, ассистент и «палочка» сообщают, что
> недоступны). Если SMTP недоступен — форма вернёт ошибку, и пользователь это увидит.

## API

| Метод | Путь | Назначение |
|---|---|---|
| `GET` | `/api/health` | Проверка живости + флаг доступности AI |
| `POST` | `/api/contact` | Приём заявки, AI-анализ, отправка писем |

## Деплой на Ubuntu 24.04 (с предустановленным Docker)

```bash
# 1. перенести папку проекта на сервер (scp/rsync или git clone + .env)
scp -r testovoe-lending user@server:/opt/

# 2. на сервере
cd /opt/testovoe-lending
docker compose up -d --build
```

Готово — сервис на `http://<ip>:41321`. Для домена/HTTPS можно поставить
впереди nginx как reverse-proxy (приложение уже учитывает `trust proxy`).

## Использование AI при разработке (честно)

Тестовое прямо просит рассказать, как использовался ИИ — рассказываю как есть.

**Какими инструментами пользовался**
- **Claude (Anthropic), Claude Code** — основной AI-напарник: скаффолдинг
  структуры, генерация черновиков модулей, SCSS и шаблонов писем, помощь с README.
- **Groq API (`openai/gpt-oss-120b`)** — AI прямо в продукте (3 функции выше).

**Что делал с помощью ИИ**
- Сгенерировал каркас фронтенда и бэкенда, болерплейт и первые версии компонентов.
- Черновики SCSS-стилей и HTML-шаблонов писем.
- Первый проход по тексту README и комментариям.

**Что правил и доводил вручную (и почему это важно)**
- **SMTP под Beget**: подобрал корректную пару `port/secure` (465 + SSL) под рабочий
  ящик, добавил таймауты и `verify()` при старте, `Reply-To` на заявителя.
- **Монорепо и `.env`**: настроил загрузку `.env` из корня и из `server/` (dotenv),
  чтобы один файл работал и в dev, и в Docker.
- **SSE-стриминг**: вручную отладил парсинг событий (буфер по `\n\n`, `[DONE]`,
  проксирование ошибок как событий, отмена через `AbortController`).
- **Лимит тела запроса**: поднял до 256kb — кириллица занимает 2 байта/символ,
  и история чата упиралась в стандартный лимит.
- **Точность контента**: описания проектов (`moimoment.ru`, `anonim.space`,
  `litra.su`) выверил по реальным сайтам, а не «как придумала модель».
- **Согласованность валидации**: привёл правила на клиенте к серверным (`zod`),
  чтобы UX-проверка и источник правды не расходились.
- **Docker**: multi-stage сборка, резолвинг пути до статики (`../public` в образе),
  healthcheck, `npm install` вместо `npm ci` (в репозитории нет lock-файла).

Подход простой: **AI пишет черновик — я проверяю факты и логику, правлю руками и
отвечаю за результат.** Контроль и понимание кода остаются за мной.
